## 四、根因分析

### 核心问题 1：知识库越权

**现象**：Agent 应该调用数据库工具但直接基于知识库回答

**原因**：`create_react_agent` 的 system prompt 中虽然提到了"系统会自动生成 SQL 并执行"，但没有**强制要求优先使用工具**。当知识库中有相关政策文本时，Agent 倾向用知识库回答。

**影响用例**：#1、#4、#7、#15、#16、#20、#21、#22（共 8 例）

**修复方向**：System Prompt 开头加 "**所有数据查询问题必须优先调用 query_database 工具，不得使用知识库回答**"

---

### 核心问题 2：数据库路由错误

**现象**：`wa_insurances_fund_month_detail` 在薪酬库，但部门/人员查询被路由到人事库

**原因**：`_is_salary_question()` 基于关键词（工资、薪酬等）判断，但"部门""人员""归属"等不含薪酬关键词，被路由到人事库。然而社保表中的 `org_name`/`dept_name`/`psn_name` 才是正确答案口径。

**影响用例**：#9、#12、#13、#14、#17、#19（共 6 例）

**修复方向**：
- 方案A：**跨库查询**——一次查询可同时访问人事库和薪酬库
- 方案B：路由中加入更多关键词（"社保""部门归属""编码YG*"等触发薪酬库）

---

### 核心问题 3：SQL 生成覆盖不全

**现象**：某些查询类型（返聘人员、中层干部级别、跨库 JOIN）无对应 SQL 模板或示例

**原因**：
- Few-shot 示例未覆盖"退休返聘"（`on_job_class='HRHI0209'`）
- 未覆盖字典值查询（"中层干部"→`HROM0101`）
- 未覆盖薪酬库社保表的人员查询模式

**影响用例**：#2、#7、#10、#24

**修复方向**：扩充 SQL 示例库，补充返聘、字典值、社保表人员查询等模式

---

### 核心问题 4：Prompt 策略矛盾

**现象**：#11（"2026年之前入职人数"返回0）——因为 Prompt 禁止 `in_duty_date`，但此问题恰好需要

**原因**：为修复 minimax-m2.5 的幻觉问题，加了"禁止擅自加日期过滤"，但一刀切导致合法日期查询也被拦截

**修复方向**：区分"用户明确说的日期"和"模型凭空加的日期"，只在后者时剥离

---

### 核心问题 5：数据口径不一致

**现象**：同一个人数统计，人事库和薪酬库社保表给出的结果不同

**原因**：
- 人事库 `hi_psn_job`：含全部任职记录（可能一人多条）
- 薪酬库 `wa_insurances_fund_month_detail`：按实际发薪口径

**修复方向**：明确告知用户当前使用的是哪个库的数据，或统一口径

---

## 六、测试结论

当前 Text-to-SQL Agent 在**纯人事库统计类查询**（总人数、部门统计、个人信息）上基本可用，但在以下场景存在严重问题：

1. **知识库干扰**——Agent 必须先查数据库再说话
2. **跨库查询缺失**——薪酬库社保表（`wa_insurances_fund_month_detail`）承载了大量部门/人员/归属的正确答案，但未被路由覆盖
3. **SQL 模板不全**——退休返聘、字典值、社保表等模式缺失

**通过率仅 3.8%，不建议在当前状态下投入生产使用。** 需至少完成 P0 修复后重新测试。
